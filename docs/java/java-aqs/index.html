java-aqs

title: "AQS" id: AQS description: "MYSQL" slug: /docs/installation sidebar_label: "AQS" sidebar_position: 1 tags: [AQS,JAVA]​

⚙️ 一、AQS的核心原理 AQS的作用与设计思想 AQS是Java并发包的基础框架，通过 状态变量（state） 和 FIFO双向队列（CLH变体） 管理线程同步。 模板方法模式：子类只需实现tryAcquire、tryRelease等方法定义资源获取规则，AQS处理线程排队、阻塞/唤醒等通用逻辑。 适用场景：ReentrantLock、Semaphore、CountDownLatch等均基于AQS实现。 核心组件解析

State（状态变量）：​

volatile int类型，通过CAS原子操作（compareAndSetState）更新。 不同子类赋予不同语义： ReentrantLock：state=0表示锁空闲，>0表示重入次数。 Semaphore：state表示剩余许可证数量。

CLH队列：​

双向链表结构，节点Node封装等待线程、等待状态（waitStatus）及前后指针。 节点状态：CANCELLED（取消）、SIGNAL（需唤醒后继节点）、CONDITION（条件等待）。 🔄 二、同步机制工作流程 资源获取（acquire） 独占模式（如ReentrantLock）：

public final void acquire(int arg) {
if (!tryAcquire(arg) &&
acquireQueued(addWaiter(Node.EXCLUSIVE), arg))
selfInterrupt();
}

步骤： tryAcquire()尝试直接获取资源（子类实现）。 失败则调用addWaiter()将线程封装为Node加入队列尾部。 acquireQueued()自旋检查是否为头节点后继，是则再次尝试获取；否则调用LockSupport.park()挂起线程。 共享模式（如Semaphore）： 通过acquireShared()唤醒后续节点形成“连锁反应”（传播唤醒），减少多次唤醒开销。 资源释放（release）

public final boolean release(int arg) {
if (tryRelease(arg)) {
Node h = head;
if (h != null && h.waitStatus != 0)
unparkSuccessor(h); // 唤醒后继节点
return true;
}
return false;
}

tryRelease()更新state，成功后检查头节点状态，唤醒队列中第一个有效节点（非CANCELLED）。

特性 公平锁 非公平锁 实现逻辑 先检查队列是否有等待线程（hasQueuedPredecessors()），有则排队 直接CAS抢锁，允许插队 性能 上下文切换多，吞吐量低 减少线程切换，吞吐量高 饥饿问题 避免饥饿 可能引发线程饥饿 源码位置 ReentrantLock.FairSync ReentrantLock.NonfairSync

为什么AQS用双向队列而非单向？ 高效处理取消节点：线程超时或中断时，双向链表可快速移除节点（O(1)），单向链表需遍历（O(n)）。 精准唤醒：释放资源时需从尾部向前遍历找到首个有效节点（避免CANCELLED节点）

AQS如何处理线程中断？ 不可中断模式（默认）：acquire()期间被中断，仅标记中断状态，线程仍阻塞直到获取资源后再响应中断。 可中断模式：acquireInterruptibly()被中断时直接抛出InterruptedException

ConditionObject与Object.wait()的区别？ 对比项 ConditionObject Object.wait() 绑定锁 必须绑定AQS实现的锁（如ReentrantLock） 任意synchronized对象 多条件队列 支持多个（如生产者和消费者分离） 仅一个等待队列 唤醒精度 signal()精确唤醒指定条件队列线程 notify()随机唤醒一个线程 共享模式下的唤醒传播是什么？ 释放资源时唤醒第一个节点，该节点获取资源后继续唤醒后续共享节点，形成连锁反应（如Semaphore），减少唤醒次数

state干啥用？—— ReentrantLock记重入次数，Semaphore记剩余许可证。 公平 vs 非公平？—— 非公平锁允许插队（性能高），公平锁严格排队（防饥饿）。 为什么用双向队列？—— 方便取消排队（比如线程超时了能快速移除）

AbstractQueuedSynchronizer AQS把复杂操作（排队、阻塞、唤醒）都封装好了。开发者只需实现几个简单方法： tryAcquire()：怎么抢资源？ tryRelease()：怎么释放资源？

/** waitStatus value to indicate successor's thread needs unparking */
static final int SIGNAL    = -1;

static final int CONDITION = -2;

static final int PROPAGATE = -3;
public final void acquire(int arg) {
    if (!tryAcquire(arg) &&
            acquireQueued(addWaiter(Node.EXCLUSIVE), arg)) {
        selfInterrupt(); // 补偿中断

// 将线程封任务封装成为节点放在队列末尾
// 将节点的前指针指向tail，然后竞争tail指向自己（死循环）
// 如果没有头节点，则将自己设置成头节点
private Node addWaiter(Node mode) {
    Node node = new Node(Thread.currentThread(), mode);
    // Try the fast path of enq; backup to full enq on failure
    Node pred = tail;
    if (pred != null) {
        node.prev = pred;
        if (compareAndSetTail(pred, node)) {
            pred.next = node;
            return node;
        }
    }
    enq(node);
    return node;
}
private Node enq(final Node node) {
    for (;;) {
        Node t = tail;
        if (t == null) { // Must initialize
            if (compareAndSetHead(new Node()))
                tail = head;
        } else {
            node.prev = t;
            if (compareAndSetTail(t, node)) {
                t.next = node;
                return t;
            }
        }
    }
}

//现线程在同步队列中的阻塞-唤醒机制
final boolean acquireQueued(final Node node, int arg) {
    boolean failed = true;
    try {
        boolean interrupted = false;
        for (;;) {
            final Node p = node.predecessor(); // 当前节点的前节点
            if (p == head && tryAcquire(arg)) { // 前驱是头节点且抢锁成功， 设置自己是头节点
                setHead(node);
                p.next = null; // help GC
                failed = false;
                return interrupted;
            }
            if (shouldParkAfterFailedAcquire(p, node) &&  // 非头节点阻塞
                    parkAndCheckInterrupt())
                interrupted = true; // 标记中断状态
        }
    } finally {
        if (failed)
            cancelAcquire(node);
    }
    
/**
 * 状态安全保证：
 只有在明确知道前驱节点会通知（SIGNAL）时才阻塞线程
 避免线程被永久阻塞（死锁风险）
 2.
 链表维护：
 自动清理已取消的节点（CANCELLED），保持链表健康
 保证等待队列中只有真正需要被唤醒的线程
 3.
 CAS操作：
 安全更新节点状态，避免同步问题
 使用 compareAndSetWaitStatus确保状态变更的原子性
 4.
 性能优化：
 优先尝试直接设置 SIGNAL 状态（避免不必要的阻塞）
 立即重试机制（返回false时）增加获取锁的机会
 */
private static boolean shouldParkAfterFailedAcquire(Node pred, Node node) {
    int ws = pred.waitStatus;
    // 前驱已设置 SIGNAL，当前线程可安全阻塞
    if (ws == Node.SIGNAL)
        /*
         * This node has already set status asking a release
         * to signal it, so it can safely park.
         */
        return true;
    // 前驱为 CANCELLED，跳过无效节点   
    if (ws > 0) { // 节点因超时/中断被取消
        /*
         * Predecessor was cancelled. Skip over predecessors and
         * indicate retry.
         */
        // 当前节点是取消状态，则从后往前遍历节点,直到拿到一个小于0的节点退出，为什么这里没做线程安全处理 
        // 这里因为是修改当前节点和前节点，不会影响后面的节点， 节点的插入已经在addWaiter中处理好了，这里仅仅是清理没用的节点
        do {
            node.prev = pred = pred.prev;
        } while (pred.waitStatus > 0);
        pred.next = node;
    } else {
        /*
         * waitStatus must be 0 or PROPAGATE.  Indicate that we
         * need a signal, but don't park yet.  Caller will need to
         * retry to make sure it cannot acquire before parking.
         */
        // 前驱状态为 0 或 PROPAGATE，CAS 更新为 SIGNAL
        compareAndSetWaitStatus(pred, ws, Node.SIGNAL);
    }
    return false;
}
}


private void cancelAcquire(Node node) {
    // Ignore if node doesn't exist
    if (node == null)
        return;

    node.thread = null;

    // Skip cancelled predecessors
    // 2. 跳过已取消的前驱
    Node pred = node.prev;
    while (pred.waitStatus > 0)
        node.prev = pred = pred.prev;

    // predNext is the apparent node to unsplice. CASes below will
    // fail if not, in which case, we lost race vs another cancel
    // or signal, so no further action is necessary.
    Node predNext = pred.next;

    // Can use unconditional write instead of CAS here.
    // After this atomic step, other Nodes can skip past us.
    // Before, we are free of interference from other threads.
    node.waitStatus = Node.CANCELLED;

    // 自己是尾节点， 则清理自己即可
    // If we are the tail, remove ourselves.
    if (node == tail && compareAndSetTail(node, pred)) {
        compareAndSetNext(pred, predNext, null);
    } else {
        // If successor needs signal, try to set pred's next-link
        // so it will get one. Otherwise wake it up to propagate.
        int ws;
        // 节点在中间，（非头部节点）， 清理掉自己，将自己的前后进行相连
        if (pred != head &&
                ((ws = pred.waitStatus) == Node.SIGNAL ||
                        (ws <= 0 && compareAndSetWaitStatus(pred, ws, Node.SIGNAL))) &&
                pred.thread != null) {
            Node next = node.next;
            if (next != null && next.waitStatus <= 0)
                compareAndSetNext(pred, predNext, next);
        } else {
            //  特殊处理：唤醒后继防止阻塞
            unparkSuccessor(node);
        }

        node.next = node; // help GC
    }
}

aqs 资源锁定方法 tryAcquire 自己实现 线程节点有几种状态: 头节点的后驱节点， 在队列的其他节点， 等待入队的节点

final int acquire(Node node, int arg, boolean shared,
                      boolean interruptible, boolean timed, long time) {
        Thread current = Thread.currentThread();
        byte spins = 0, postSpins = 0;   // retries upon unpark of first thread
        boolean interrupted = false, first = false;
        Node pred = null;                // predecessor of node when enqueued

        /*
         * Repeatedly:
         *  Check if node now first
         *    if so, ensure head stable, else ensure valid predecessor
         *  if node is first or not yet enqueued, try acquiring
         *  else if node not yet created, create it
         *  else if not yet enqueued, try once to enqueue
         *  else if woken from park, retry (up to postSpins times)
         *  else if WAITING status not set, set and retry
         *  else park and clear WAITING status, and check cancellation
         */
        // 自旋
        for (;;) {
            // first ：头节点（head）的直接后继节点
            // 这里是从后往前进行遍历的
            if (!first && (pred = (node == null) ? null : node.prev) != null &&
                !(first = (head == pred))) {
                // 前驱节点已取消，清理队列
                if (pred.status < 0) {
                    cleanQueue();           // predecessor cancelled
                    continue;
                } else if (pred.prev == null) {
                    Thread.onSpinWait();     // 前驱节点正在修改，自旋等待稳定 ？？
                    continue;
                }
            }
            // 当前节点为头节点
            if (first || pred == null) {
                boolean acquired;
                try {
                    // cas获取锁标识
                    if (shared)
                        acquired = (tryAcquireShared(arg) >= 0);
                    else
                        acquired = tryAcquire(arg);
                } catch (Throwable ex) {
                    cancelAcquire(node, interrupted, false); // 异常时取消获取
                    throw ex;
                }
                // 获取锁成功后，当前节点设置为头节点
                if (acquired) {
                    if (first) {
                        node.prev = null;
                        head = node;
                        pred.next = null;
                        node.waiter = null;
                        if (shared)
                            signalNextIfShared(node); // 共享模式唤醒后续节点
                        if (interrupted)
                            current.interrupt();  // 恢复中断状态
                    }
                    return 1;
                }
            }
            // 初始化节点
            if (node == null) {                 // allocate; retry before enqueue
                if (shared)
                    node = new SharedNode();
                else
                    node = new ExclusiveNode();
            } else if (pred == null) {          // try to enqueue
                // 前驱节点为null，说明还没入队， 尝试入队
                node.waiter = current;  // 绑定当前线程
                Node t = tail;
                node.setPrevRelaxed(t);         // 设置前驱节点
                if (t == null)
                    tryInitializeHead(); // 没有节点，则尝试初始化节点 ，cas初始化节点
                else if (!casTail(t, node)) // cas比较入队
                    node.setPrevRelaxed(null);  // back out
                else
                    t.next = node;      //  成功入队
            } else if (first && spins != 0) {
                --spins;                        // 减少自旋次数，避免过度不公平
                Thread.onSpinWait();
            } else if (node.status == 0) {
                node.status = WAITING;          // 标记节点为等待唤醒状态
            } else {
                // 阻塞线程（支持超时）
                long nanos;
                spins = postSpins = (byte)((postSpins << 1) | 1);
                if (!timed)
                    LockSupport.park(this);
                else if ((nanos = time - System.nanoTime()) > 0L)
                    LockSupport.parkNanos(this, nanos);
                else
                    break;
                node.clearStatus();
                if ((interrupted |= Thread.interrupted()) && interruptible)
                    break;
            }
        }
        //取消调度 最上面的elese 逻辑可知。超时获取失败或者线程打断后就会触发该逻辑
        return cancelAcquire(node, interrupted, interruptible); 
    }

    private static void signalNext(Node h) {
        Node s;
        if (h != null && (s = h.next) != null && s.status != 0) {
            s.getAndUnsetStatus(WAITING);
            LockSupport.unpark(s.waiter);
        }
    }

无效节点清理


private void cleanQueue() {
        
        for (;;) {                               // restart point
            // q当前系节点， s后继节点， p：前驱节点
            // 从后往前遍历
            for (Node q = tail, s = null, p, n;;) { // (p, q, s) triples
                if (q == null || (p = q.prev) == null)
                    return;                      // end of list
                if (s == null ? tail != q : (s.prev != q || s.status < 0))
                    break;                       // inconsistent
                if (q.status < 0) {              // cancelled
                    // 为什么这里先设置前驱节点，在设置后驱节点？，保证从后到前的队列不会断裂
                    if ((s == null ? casTail(q, p) : s.casPrev(q, p)) &&
                        q.prev == p) {
                        p.casNext(q, s);         // OK if fails  
                        if (p.prev == null) // 没有前驱节点就是头节点了，需要唤醒后续的节点
                            signalNext(p);
                    }
                    break;
                }
                if ((n = p.next) != q) {         // help finish
                    if (n != null && q.prev == p) {
                        p.casNext(n, q);
                        if (p.prev == null)
                            signalNext(p);
                    }
                    break;
                }
                s = q;
                q = q.prev;
            }
        }
    }

ReentrantLock

非公平锁实现 在调用加锁逻辑后先尝试获取锁，而不是直接入队列 当前线程直接重入 java.util.concurrent.locks.ReentrantLock.NonfairSync

    final boolean initialTryLock() {
        Thread current = Thread.currentThread();
        if (compareAndSetState(0, 1)) { // first attempt is unguarded
            setExclusiveOwnerThread(current);
            return true;
        } else if (getExclusiveOwnerThread() == current) {
            int c = getState() + 1;
            if (c < 0) // overflow
                throw new Error("Maximum lock count exceeded");
            setState(c);
            return true;
        } else
            return false;
    }

公平锁实现 先查看队列里面是否还存state > 0的节点,如果不存在则直接先进行锁的获取 hasQueuedThreads这个在AQS中实现了，而且是从后往前遍历的（这是为什么） 当前线程直接重入 java.util.concurrent.locks.ReentrantLock.FairSync

    final boolean initialTryLock() {
        Thread current = Thread.currentThread();
        int c = getState();
        if (c == 0) {
            if (!hasQueuedThreads() && compareAndSetState(0, 1)) {
                setExclusiveOwnerThread(current);
                return true;
            }
        } else if (getExclusiveOwnerThread() == current) {
            if (++c < 0) // overflow
                throw new Error("Maximum lock count exceeded");
            setState(c);
            return true;
        }
        return false;
    }
